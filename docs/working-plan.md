# 金融资产问答系统详细执行计划

## 1. 执行目标

本计划用于指导项目从空仓库推进到可演示版本。目标不是一次性做大，而是按阶段交付，每个阶段都具备清晰目标、具体产出和验证方案。

总体原则：

- 每个阶段都必须有可运行成果
- 每个阶段都必须有验收标准
- 先打通主链路，再补能力
- 先保证结构，再优化效果

## 2. 里程碑划分

建议按 6 个阶段推进。

### 阶段 0：项目初始化

目标：

- 建立前后端基础目录
- 建立最小可运行开发环境
- 明确配置文件和依赖管理方式

主要工作：

- 初始化 `frontend/`，创建 React + TypeScript + Vite 项目
- 初始化 `backend/`，创建 FastAPI 项目结构
- 新建基础文档：README、架构文档、执行计划
- 增加环境变量模板，如 `.env.example`
- 约定统一代码风格和命名规范

交付物：

- 可启动的前端工程
- 可启动的后端工程
- 仓库文档骨架

验证计划：

- 运行前端，确认首页可打开
- 运行后端，确认 `/health` 返回成功
- 检查前后端目录是否符合方案设计

验收标准：

- 前后端都能本地启动
- 仓库结构不混乱
- 后续开发所需基础设施已具备

### 阶段 1：后端主链路骨架

目标：

- 建立统一请求与响应协议
- 建立问答主入口
- 建立路由、服务、工具的基础分层

主要工作：

- 定义 `POST /api/v1/chat`
- 定义 Pydantic 请求与响应模型
- 实现 `router_service` 基础版本
- 实现空的 `asset_qa_service` 与 `knowledge_qa_service`
- 实现统一错误返回格式

交付物：

- 可调用的问答主接口
- 统一 JSON 响应模型
- 后端主链路骨架代码

验证计划：

- 用固定 mock 输入调用 `/chat`
- 检查不同意图是否能被正确路由
- 检查非法参数是否被 Pydantic 拦截

验收标准：

- `/chat` 可接收请求并返回合法结构
- 路由逻辑可区分资产类与知识类问题
- 错误处理路径清晰

### 阶段 2：资产问答能力

目标：

- 支持当前价格查询
- 支持 7 天、30 天涨跌分析
- 支持基础趋势总结

主要工作：

- 封装 `MarketDataTool`
- 接入 `yfinance`
- 实现价格历史数据获取
- 实现涨跌幅计算逻辑
- 输出基础结构化结论

交付物：

- 可回答“当前股价是多少”
- 可回答“最近 7 天涨跌如何”
- 可为前端提供走势图数据

验证计划：

- 用 `BABA`、`TSLA` 等样例做接口测试
- 检查价格与涨跌计算是否和原始数据一致
- 检查无效 ticker 是否能稳定报错或给出提示

验收标准：

- 返回结果中数值字段完整
- `summary` 与 `objective_data` 不冲突
- 趋势接口可供前端绘图

### 阶段 3：RAG 知识问答能力

目标：

- 建立小型金融知识库
- 支持金融概念问答
- 支持基于财报摘要的回答

主要工作：

- 设计知识库目录与文档格式
- 实现 ingest 脚本
- 实现切块、嵌入、向量存储
- 实现 `retriever`
- 实现 `knowledge_qa_service`

交付物：

- 可导入知识文档
- 可执行检索
- 可回答基础金融知识问题

验证计划：

- 用“什么是市盈率”“收入和净利润的区别是什么”测试
- 检查回答是否引用实际知识库来源
- 对不存在知识的问题测试“依据不足”路径

验收标准：

- 检索结果能够命中正确片段
- 回答不脱离检索上下文
- 未命中时能明确说明信息不足

### 阶段 4：Prompt、生成与校验层

目标：

- 固化 Prompt 模板职责
- 建立统一结构化生成机制
- 建立回答验证机制

主要工作：

- 实现意图路由 Prompt
- 实现检索改写 Prompt
- 实现回答生成 Prompt
- 实现输出解析器和校验器
- 增加 `verification_service`

交付物：

- Prompt 模板文件
- 输出 JSON parser
- 结构校验与一致性校验逻辑

验证计划：

- 构造多类问题，检查输出结构是否稳定
- 人工对比数值字段与生成摘要是否一致
- 对缺少来源、字段错误、无依据分析的情况做异常测试

验收标准：

- 回答结构稳定可解析
- 不合格输出可被识别并拒绝或降级
- 客观数据与分析字段边界清晰

当前实现状态：

- 已完成
- 已接入 `query_rewrite_service`、`answer_generation_service`、`verification_service`
- 已支持 `ollama` 与 `openai` 两类 LLM provider
- 已加入“依据不足时禁止自由生成”的确定性约束
- 已验证中文概念词条可回退到英文官方词条来源

### 阶段 5：前端 MVP

目标：

- 提供一个清晰的演示页面
- 展示结构化回答
- 展示价格走势

主要工作：

- 实现问答输入框
- 实现回答卡片组件
- 实现来源与限制说明区域
- 实现价格图表组件
- 打通前后端调用

交付物：

- 单页演示前端
- 结构化问答展示界面
- 历史走势可视化

验证计划：

- 手工输入价格问题和知识问题
- 检查页面是否正确渲染不同回答区块
- 检查加载、错误、空状态是否清晰

验收标准：

- 页面可完成端到端问答
- 图表和结构化回答同步展示
- 演示路径简洁，不需要复杂操作

当前实现状态：

- 已完成
- 已实现单页前端 MVP
- 已接入 `/api/v1/chat` 与 `/api/v1/assets/{symbol}/history`
- 已实现结构化回答展示、来源与限制说明展示、资产走势图展示
- 已验证 Vite 代理转发到后端可正常工作

### 阶段 6：联调、测试与交付整理

目标：

- 稳定主流程
- 补齐关键测试
- 形成可提交作业版本

主要工作：

- 联调前后端
- 增加后端单元测试和接口测试
- 补充示例问题和演示截图
- 完善 README 与架构说明
- 整理已知限制与后续扩展方向

交付物：

- 稳定可运行版本
- 测试与演示文档
- 可直接交付的仓库

验证计划：

- 完整跑通至少 5 个演示问题
- 检查 API 返回、前端展示、日志输出是否一致
- 检查启动步骤是否能按 README 重现

验收标准：

- 评审者能在较少步骤内启动并体验系统
- 主要功能无阻塞问题
- 文档足以说明设计思路和取舍

## 4. 每阶段测试清单

建议至少准备以下验证样例：

### 资产类

- `阿里巴巴当前股价是多少？`
- `BABA 最近 7 天涨跌情况如何？`
- `特斯拉最近 30 天走势如何？`

### 知识类

- `什么是市盈率？`
- `收入和净利润的区别是什么？`

### 财报类

- `某公司最近季度财报摘要是什么？`

### 异常类

- `XXXX123 当前股价是多少？`
- `请预测某股票未来 30 天走势`
- `知识库中没有覆盖的冷门问题`

验证重点：

- 是否正确路由
- 是否调用正确工具
- 是否有来源引用
- 是否明确限制与不确定性

## 5. 风险缓冲计划

若阶段推进中出现问题，建议按以下方式降级：

- 若新闻数据不稳定，先不做事件归因，只保留“可能因素”并注明不确定性
- 若 RAG 效果不稳，先缩小知识库范围，提高命中率
- 若 LLM 输出结构不稳定，先使用更强的 schema 约束，必要时增加后处理修正
- 若前端时间不足，只保留单页输入、结果卡片和折线图

## 6. 完成定义

项目达到以下条件时，可视为完成可交付版本：

- 用户可以在前端输入问题并获得回答
- 至少支持资产价格/走势问答和基础金融知识问答
- 回答为结构化展示，不是纯自由文本
- 回答包含客观数据、分析说明、来源或依据提示
- 系统具备基本校验和错误处理
- 文档能够清楚说明架构、Prompt 思路、RAG 设计和验证方法
